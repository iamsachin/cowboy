    **COST-01 fix (analytics.ts ~line 119):**
    In `computePeriodStats`, change the `perConvTokens` query's `.groupBy(tokenUsage.conversationId)` to `.groupBy(tokenUsage.conversationId, tokenUsage.model)`. This ensures each model's tokens are priced separately via `calculateCost()`. The existing loop already calls `calculateCost(row.model, ...)` per row and sums costs, so adding the model to groupBy is the only change needed.

    **COST-02 fix (analytics.ts ~line 306):**
    In `getConversations`, replace the cost sort column from `sql\`sum(inputTokens)\`` to a subquery that computes actual cost. The subquery should:
    1. Join token_usage grouped by model
    2. Apply per-model pricing using a CASE expression matching model names to their inputPerMTok/outputPerMTok rates from MODEL_PRICING
    3. Sum the computed costs

    Alternative simpler approach: Since `getConversations` already computes `costResult` per row (line 347), sort the rows in JavaScript after fetching. Change the SQL sort to use `conversations.createdAt` as default when sort='cost', then after mapping rows to `conversationRows`, sort by `cost` field in JS. This avoids duplicating pricing logic in SQL.

    Choose the simpler JS sort approach: fetch with default date sort, then re-sort in JS by cost. Adjust the offset/limit handling — for cost sort, fetch all matching rows, sort by cost in JS, then slice for pagination.

    Actually, the cleanest approach: add a computed cost column or use a subquery. Since the dataset is small (hundreds of conversations, not millions), the JS approach is fine:
    - When `sort === 'cost'`: remove LIMIT/OFFSET from SQL, sort `conversationRows` by `.cost` (nulls last), then slice for pagination
    - Keep SQL sort for all other columns

    **COST-06:** Add a `formatCost()` function to `format-tokens.ts` that will be the single source of truth for all cost displays.

    The function should handle all cost ranges with appropriate precision:
    ```typescript
    export function formatCost(cost: number): string {
      if (cost === 0) return '$0.00';
      if (cost >= 0.01) return `$${cost.toFixed(2)}`;
      if (cost >= 0.001) return `$${cost.toFixed(3)}`;
      if (cost >= 0.0001) return `$${cost.toFixed(4)}`;
      return '< $0.0001';
    }
    ```

    This replaces the various inline `toFixed(2)`, `Intl.NumberFormat` currency formatters, and local `formatCost()` functions scattered across components. The existing `formatTurnCost()` can remain as an alias or be deprecated — but `formatCost()` is the canonical version.

    Note: This task only creates the utility. Plan 02 will wire it into all components.
