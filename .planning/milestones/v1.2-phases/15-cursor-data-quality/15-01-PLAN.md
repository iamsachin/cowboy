    - Test: Assistant bubble with empty text, no capability flags, and no toolFormerData still produces message (backward compat -- already passes)
    - Test: Workspace path extracted from composerData produces last directory segment as project name
    - Test: Missing workspace path falls back to 'Cursor' as project name

  <action>
    **cursor-parser.ts changes:**
    1. In `parseCursorDb()`, extract workspace context from the composerData JSON. Investigate fields like `data.context`, `data.workspacePath`, `data.workspaceFolder`, or `data.rootDir`. The Cursor DB path itself (`~/.cursor/User/workspaceStorage/{hash}/state.vscdb`) is the globalStorage DB, not per-workspace -- so workspace info must come from composerData fields.
    2. Add a `workspacePath` field (string | null) to the `CursorConversation` interface. Parse it from whichever field contains workspace context in composerData.
    3. If composerData does not contain workspace path info, leave as null.

    **cursor-normalizer.ts changes:**
    1. Fix the assistant bubble skip logic (lines 55-58): Currently skips bubbles with `isCapabilityIteration || toolFormerData` when text is empty. Instead of skipping entirely, generate a tool activity summary. Count capability iterations per conversation and produce content like "Executed N tool calls" for empty-text capability bubbles that are adjacent (group them).
    2. Actually, simpler approach per user decision: keep the current skip for pure tool execution bubbles BUT ensure that assistant bubbles with actual text content are NEVER skipped regardless of `isCapabilityIteration` flag. The existing code already does this (line 55 checks `!bubble.text?.trim()`), so the "Empty response" issue likely comes from bubbles where `text` is empty string but the actual content lives in another field.
    3. Investigate: In the parser, also extract `data.richText`, `data.rawText`, `data.codeBlocks`, or any other content field from bubble data. Add these to `CursorBubble` interface if found.
    4. In the normalizer, when `bubble.text` is empty/null, check these alternative content fields and use the first non-empty one.
    5. For genuinely empty assistant bubbles (pure tool execution with no text in any field), produce content like "Executed tool call" with the capability type if available, instead of null content. This prevents "Empty response" in the UI.

    **index.ts changes (line 188):**
    1. Replace hardcoded `'Cursor'` with workspace-derived project name.
    2. Extract last directory segment from `conv.workspacePath` using the same pattern as `deriveProjectName()` from `file-discovery.ts` (take basename / last path segment).
    3. Fallback to `'Cursor'` if `workspacePath` is null.

    **Important:** The `data.text` field in bubble JSON may be the issue -- the parser currently does `data?.text ?? ''` which defaults to empty string. Check if Cursor stores content under a different key for some bubble types. The executor should log/inspect a sample bubble JSON to understand the actual structure if tests alone are insufficient.
