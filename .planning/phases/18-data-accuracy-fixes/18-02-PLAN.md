**DATA-05: Tool durations**

All tool call durations are `null` because JSONL format doesn't include execution time. The Avg Duration and P95 Duration columns in ToolStatsTable always show "N/A".

Two options: remove the columns or estimate from message timestamps. Since tool calls are linked to messages via messageId, and messages have timestamps, we can't reliably estimate individual tool call duration when multiple tool calls occur in the same assistant message.

Remove the duration columns from the tool stats display:

1. In `packages/frontend/src/components/ToolStatsTable.vue`, remove the `avgDuration` and `p95Duration` columns from the `columns` array (lines 54-55). Keep only: Tool Name, Calls, Success Rate.

2. Remove the corresponding `<td>` elements for avgDuration and p95Duration from the template (lines 33-34).

3. Remove the `formatDuration` helper function (lines 109-112) since it's no longer used.

4. In `packages/backend/src/db/queries/analytics.ts` `getToolStats` function (line 626-672): Remove the `avgDuration` computation from the main query (line 640) and remove the entire P95 secondary query loop (lines 650-671). Simplify the return to omit avgDuration and p95Duration. Set them to `null` in the return for API compatibility (the type still expects them).

Actually, for backward compatibility, keep the fields in `ToolStatsRow` type but just stop computing P95 (expensive secondary query per tool). Set avgDuration and p95Duration to `null` in the response. The frontend columns are already removed so they won't display.

**DATA-06: Null-status tool calls**

In `packages/backend/src/db/queries/analytics.ts` `getToolStats` function, the SQL currently counts ALL rows in `total` (line 637), but only counts known statuses in success/failure. 427 tool calls with `status = NULL` inflate the denominator.

Change the success rate computation to exclude null-status calls from the denominator. Add an `unknown` count:

Replace the select fields (lines 637-639):
```sql
total: sql<number>`count(*)`,
success: sql<number>`sum(case when ${toolCalls.status} = 'success' or ${toolCalls.status} = 'completed' then 1 else 0 end)`,
failure: sql<number>`sum(case when ${toolCalls.status} is not null and ${toolCalls.status} != 'success' and ${toolCalls.status} != 'completed' then 1 else 0 end)`,
```

With:
```sql
total: sql<number>`count(*)`,
success: sql<number>`sum(case when ${toolCalls.status} = 'success' or ${toolCalls.status} = 'completed' then 1 else 0 end)`,
failure: sql<number>`sum(case when ${toolCalls.status} is not null and ${toolCalls.status} != 'success' and ${toolCalls.status} != 'completed' then 1 else 0 end)`,
unknown: sql<number>`sum(case when ${toolCalls.status} is null then 1 else 0 end)`,
```

Update `ToolStatsRow` in `packages/shared/src/types/api.ts` to add `unknown: number`.

In `packages/frontend/src/components/ToolStatsTable.vue`, change `getSuccessRate` to use `(total - unknown)` as denominator:

```typescript
  return (row.success / denominator) * 100;
}
```

Add the `unknown` field to the return in the analytics query's map function.

**DATA-07: ExitPlanMode user rejections**

All 44 ExitPlanMode tool calls have `is_error: true` in the JSONL, causing them to be classified as errors. But ExitPlanMode with is_error=true means the user rejected the plan, not that the tool failed.

In `packages/backend/src/ingestion/normalizer.ts`, in the tool call creation loop (lines 142-158), add special handling for ExitPlanMode:

Change the status assignment (line 154):
```typescript
status: result ? (result.isError ? 'error' : 'success') : null,
```

To:
```typescript
status: result
  ? (toolUse.name === 'ExitPlanMode' && result.isError
    ? 'rejected'
    : result.isError ? 'error' : 'success')
  : null,
```

This classifies ExitPlanMode with is_error as 'rejected' instead of 'error'.

Then in `packages/backend/src/db/queries/analytics.ts` `getToolStats`, update the success/failure SQL to treat 'rejected' as neither success nor failure (excluded from both):

```sql
success: sql<number>`sum(case when ${toolCalls.status} = 'success' or ${toolCalls.status} = 'completed' then 1 else 0 end)`,
failure: sql<number>`sum(case when ${toolCalls.status} = 'error' then 1 else 0 end)`,
unknown: sql<number>`sum(case when ${toolCalls.status} is null then 1 else 0 end)`,
rejected: sql<number>`sum(case when ${toolCalls.status} = 'rejected' then 1 else 0 end)`,
```

Update `ToolStatsRow` type to include `rejected: number`.

In the frontend ToolStatsTable, the success rate should use `(total - unknown - rejected)` as denominator so rejected calls don't count against success rate either:

```typescript
function getSuccessRate(row: ToolStatsRow): number {
  const denominator = row.total - (row.unknown ?? 0) - (row.rejected ?? 0);
  if (denominator === 0) return 0;
  return (row.success / denominator) * 100;
}
```

Note: The 'rejected' classification only applies on re-ingestion. Existing data will still have 'error' status until the database is re-ingested. This is acceptable since the user can trigger re-ingestion from Settings.
