**DATA-08: Heatmap timezone fix**

In `packages/frontend/src/components/ActivityHeatmap.vue`, the `cells` computed (lines 71-110) creates Date objects with `new Date(dates[0] + 'T00:00:00')` (line 82-83) which uses LOCAL timezone, but then formats using `toISOString().slice(0, 10)` (line 101) which converts to UTC. For UTC-negative timezones, this shifts dates back by one day.

Fix the date iteration to stay in local timezone throughout:

1. Change lines 82-83 to not append timezone suffix (these are fine -- `T00:00:00` without Z is local):
```typescript
const minDate = new Date(dates[0] + 'T00:00:00');
const maxDate = new Date(dates[dates.length - 1] + 'T00:00:00');
```
These are already correct -- they parse as local time.

2. The bug is on line 101: `current.toISOString().slice(0, 10)` converts to UTC. Replace with a local date formatter:

```typescript
// Replace: const dateStr = current.toISOString().slice(0, 10);
const dateStr = `${current.getFullYear()}-${String(current.getMonth() + 1).padStart(2, '0')}-${String(current.getDate()).padStart(2, '0')}`;
```

This keeps dates in local timezone, matching the server-side `date()` SQLite function output.

3. Also verify the `startDate` and `endDate` alignment operations (lines 86-95) use local-time methods (`getDay()`, `setDate()`) which they already do -- these are fine.

**DATA-09: Timeseries timestamp consistency**

In `packages/backend/src/db/queries/analytics.ts` `getTimeSeries` function (lines 154-221):

The date filter uses `conversations.createdAt` (lines 161-165) but the grouping uses `tokenUsage.createdAt` for the period (line 170). When a conversation's createdAt differs from its token usage timestamps (e.g., long-running conversations, or Cursor timestamp issues), tokens may be attributed to a different time period than expected.

Fix by grouping on `conversations.createdAt` instead of `tokenUsage.createdAt`:

Change line 170:
```typescript
period: sql<string>`strftime('${sql.raw(dateFormat)}', ${tokenUsage.createdAt})`.as('period'),
```

To:
```typescript
period: sql<string>`strftime('${sql.raw(dateFormat)}', ${conversations.createdAt})`.as('period'),
```

This ensures the grouping period matches the filter source. Both now use `conversations.createdAt` -- the canonical timestamp for when the conversation happened.

Also update the heatmap backend query `getHeatmapData` (line 688) to verify it already uses `conversations.createdAt` for the `date()` function -- it does (`date(${conversations.createdAt})`), so no change needed there.
