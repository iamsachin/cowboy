  <action>
    1. Port snapshot logic to `snapshot.rs`:
       - ConversationSnapshot struct: exists, message_count, tool_call_count, token_usage_count, plan_count, status, title, model
       - `pub fn snapshot_conversation(conn: &rusqlite::Connection, conversation_id: &str) -> ConversationSnapshot`: query conversations table, then COUNT(*) from messages/tool_calls/token_usage/plans
       - `pub fn track_changes(conn: &rusqlite::Connection, conversation_id: &str, normalized: &NormalizedData, snapshot: &ConversationSnapshot) -> Vec<serde_json::Value>`:
         If !snapshot.exists: return vec with conversation:created event
         Else: compare new counts to snapshot counts, check status/title/model changes
         Change types: "messages-added", "tool-calls-added", "tokens-updated", "plan-updated", "status-changed", "metadata-changed"
         Return vec of event payloads

    2. Port migration to `migration.rs`:
       - `pub fn run_data_quality_migration(conn: &rusqlite::Connection) -> MigrationResult`
       - Port fix_conversation_titles: find conversations with bad titles, recompute from user messages using should_skip_for_title
       - Port fix_conversation_models: fix NULL model from token_usage frequency, fix "default" model for cursor
       - Port fix_cursor_projects: read workspace paths from vscdb, update "Cursor" project names
       - Port fix_cursor_message_content: fix null/empty assistant content in cursor conversations
       - Port fix_duplicate_content_blocks: strip system XML tags from existing assistant content
       - Port clear_stale_subagent_links: one-time migration with tracking table
       - All operations use raw rusqlite SQL (not Drizzle) matching the existing Rust backend pattern

    3. Build orchestrator in `mod.rs`:
       - IngestionStatus with Arc<Mutex<>> for shared state
       - `pub async fn run_ingestion(state: &AppState) -> Result<IngestionStats>`:

         a. Ensure ingested_files table exists (CREATE TABLE IF NOT EXISTS)
         b. Run data quality migration (non-fatal)
         c. Claude Code ingestion loop:
            - discover_jsonl_files(None) to find all .jsonl files
            - For each file: check mtime/size against ingested_files cache, skip unchanged
            - parse_jsonl_file -> normalize_conversation
            - Inside db.call() closure: snapshot_conversation, transaction with all inserts (conversations ON CONFLICT UPDATE, messages/tool_calls/token_usage/compaction_events INSERT OR IGNORE), insert_extracted_plans, track_changes
            - Record file in ingested_files
            - Emit WebSocket events via broadcast_event

         d. Subagent linking (post-processing):
            Phase A: Filesystem parent linking (set parent_conversation_id)
            Phase B: Tool-call matching via link_subagents
            Update tool_calls with subagent_conversation_id and subagent_summary

         e. Cursor ingestion loop:
            - discover_cursor_db() to find vscdb
            - parse_cursor_db -> for each conv: get_bubbles_for_conversation -> normalize_cursor_conversation
            - Same DB insert pattern as Claude Code (snapshot, transaction, track_changes, events)
            - Wrap sync rusqlite calls in spawn_blocking

         f. Mark stale active conversations as completed (updated_at > 5 min ago)

         g. is_conversation_ongoing function: port from Node.js (check last message role, tool calls)

         h. insert_extracted_plans function: delete existing plans for conv, extract from assistant messages, infer step completion, insert plans + plan_steps

       - HTTP routes:
         POST /api/ingest: if running return 409, else fire-and-forget run_ingestion
         GET /api/ingest/status: return current status
       - `pub fn routes() -> Router<AppState>` merged into server.rs

    4. Update server.rs:
       - Add `use crate::ingestion;` and `.merge(ingestion::routes())` to router
       - Add auto-ingest: spawn run_ingestion after server starts (after a brief delay to let the server bind)

    All DB writes inside `state.db.call(move |conn| { ... })` closures. Transactions created and committed within a single call() closure. Use `rusqlite::params![]` for parameterized queries.

  <action>
    Create `scripts/diff-ingestion.sh` that verifies ING-05 (row-level data parity between Node.js and Rust ingestion):

    1. Script structure:
       ```bash
       #!/usr/bin/env bash
       set -euo pipefail
       # Row-level SQLite diff between Node.js and Rust ingested databases
       ```

    2. Prerequisites check: verify sqlite3 is available, verify both databases exist

    3. Process:
       a. Stop both backends if running
       b. Copy original cowboy.db to a backup
       c. Create two empty databases: cowboy-node.db and cowboy-rust.db
       d. Initialize both with the schema from src-tauri/src/schema.sql
       e. Run Node.js ingestion against cowboy-node.db: use the Node.js backend's ingest endpoint (or import and call directly)
       f. Run Rust ingestion against cowboy-rust.db: use the Rust backend's POST /api/ingest endpoint
       g. For each table (conversations, messages, tool_calls, token_usage, plans, plan_steps, compaction_events):
          - Dump both as sorted JSON: `sqlite3 cowboy-node.db ".mode json" "SELECT * FROM {table} ORDER BY id"`
          - Diff the outputs
          - Report: PASS (identical) or FAIL (show first 20 diff lines)
       h. Summary: total tables, passed, failed

    4. Alternative simpler approach (if running both backends is complex):
       - Compare the existing cowboy.db (populated by Node.js) against a fresh Rust-ingested database
       - Run Rust ingestion with the same source files
       - Dump and diff specific tables

    5. The script should be runnable as: `./scripts/diff-ingestion.sh`
       Exit code 0 if all tables match, 1 if any differ

    6. Key columns to compare per table:
       - conversations: id, agent, project, title, created_at, updated_at, model, status, parent_conversation_id
       - messages: id, conversation_id, role, content (first 200 chars to avoid huge diffs), thinking (first 200 chars), created_at, model
       - tool_calls: id, message_id, conversation_id, name, status, created_at
       - token_usage: id, conversation_id, message_id, model, input_tokens, output_tokens, cache_read_tokens, cache_creation_tokens
       - plans: id, conversation_id, source_message_id, title, total_steps, completed_steps, status
       - plan_steps: id, plan_id, step_number, content, status
       - compaction_events: id, conversation_id, timestamp, tokens_before, tokens_after
