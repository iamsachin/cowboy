```rust
// project as link-attempted. This runs unconditionally after link_subagents(),
// regardless of whether any matches were found — an empty `links` vector means
// "linker ran and found nothing", which is exactly the state the frontend needs

let parent_session_ids: Vec<String> = parent_files
    .iter()
    .map(|f| f.session_id.clone())
    .collect();

    .db

        for sid in &parent_session_ids {
            let parent_conv_id = generate_id(&["claude-code", sid]);
            tx.execute(
                "UPDATE tool_calls SET subagent_link_attempted = 1 \
                 WHERE conversation_id = ?1 AND name IN ('Task', 'Agent')",
                rusqlite::params![parent_conv_id],

        tx.commit()?;
        Ok::<_, tokio_rusqlite::Error>(())
```

```typescript
import type { SubagentSummary } from '../types';

export type GhostCardState = 'running' | 'unmatched' | 'missing' | 'summary';

export interface GhostCardFlags {
  subagentSummary: SubagentSummary | null | undefined;
  subagentLinkAttempted: boolean;
  subagentConversationId: string | null | undefined;
  isActive: boolean;
}

/**
 * Classify the render state of a sub-agent ghost/summary card.
 *
 * Precedence (high -> low):
 *   1. subagentSummary present         -> 'summary'  (rich three-tier card)
 *   2. subagentConversationId present  -> 'missing'  (link written but child JSONL gone)
 *   3. subagentLinkAttempted true      -> 'unmatched' (linker ran, no match found)
 *   4. otherwise                       -> 'running'  (linker has not yet visited)
 *
 * The `isActive` field is NOT used for classification -- it is consumed by the
 * template to decide whether to pulse the 'running' icon. Running state with
 * isActive=false is valid legacy behavior for pre-migration rows; the UI stays
 * static instead of misleadingly pulsing.
 */
export function classifyGhostState(flags: GhostCardFlags): GhostCardState {
  if (flags.subagentSummary) return 'summary';
  if (flags.subagentConversationId) return 'missing';
  if (flags.subagentLinkAttempted) return 'unmatched';
  return 'running';
}
```

<edge_case_disposition>
## Old-Data Backfill — Choosing Option (a): "Document as Acceptable"

**The edge case:** Tool_calls ingested before this plan ships have `subagent_link_attempted = 0` in the DB (the column default). For such rows, the classifier returns `"running"` whenever `subagentSummary` and `subagentConversationId` are both null — regardless of whether the parent conversation is truly still active.

**Visual consequence:** The running-state card renders with `border-info/40` styling. The `Loader2` spin animation is gated on `props.isActive`, so for pre-migration rows in **inactive** conversations the icon is static — the card reads as "Subagent running..." with no pulse and no elapsed-time display (because `runningElapsed` returns `null` when `!isActive`). For pre-migration rows in **still-active** conversations, the icon pulses incorrectly for at most one ingestion cycle — the next Phase B pass will flip the flag to `1` and the state transitions to `"unmatched"` (which is the correct terminal state for a truly unmatched sub-agent).

**Why (a) beats (b) here:**

1. **The old-link-clearing migration already ran.** `migration.rs::clear_stale_subagent_links` (lines 326-367) is a one-time migration that nukes every `subagent_conversation_id` and `subagent_summary` value from old heuristic-linked rows. Current production DBs have already been re-processed by the modern three-phase linker as of the v2.0 → v3.x transition. The sub-agent link/summary columns are therefore NOT populated by the pre-IMPR-7 codebase that this plan is patching — they are populated by the current linker, which is about to be modified by Task 2 to also set the flag. So the "stale pre-migration row" surface area is small: tool_calls that were ingested by the current linker BEFORE this plan ships.
2. **The linker re-runs on every ingestion cycle.** `link_subagents_post_processing` is invoked from `run_ingestion` after file discovery (`mod.rs:214-217`). Any subsequent ingestion pass (triggered by new JSONL writes, watcher events, or restart) re-walks every parent file in every project and will set the flag for those rows — no manual backfill needed, just the natural cadence of live usage.
3. **The user-visible defect of option (a) is narrow.** A stale inactive parent with a genuinely unmatched Task tool_call renders as "running" with no pulse. Given: (a) pre-migration rows only exist for conversations ingested before the next watch cycle, (b) the cosmetic error is strictly weaker than the current state (both old and new render a ghost card — the new one just has slightly misleading copy for a narrow window), and (c) the fix arrives automatically on the next ingestion pass, the fix-forward cost is near-zero and the defect window is small.
4. **Option (b) has concrete costs.** A one-off backfill would need (i) a new entry in `migrations_applied` to mark it done, (ii) a table scan (`SELECT id, conversation_id, name, created_at FROM tool_calls WHERE name IN ('Task', 'Agent')` with `subagent_link_attempted = 0`), (iii) a heuristic to decide "parent conversation's ingestion completed before this migration" — which is not actually tracked anywhere. `ingested_files.ingested_at` exists but isn't keyed to conversation_ids; mapping would require joining through `session_id` extracted from `file_path`. None of this is infeasible, but it's ~40 lines of migration Rust to solve a cosmetic transient for a column whose correct value arrives naturally on the next ingestion.

**Option (a) remediation recipe (for any user who reports this):** "Open any active Claude Code session, let it ingest one cycle. Cards will flip from 'running' to 'unmatched' if the linker made a decision." Or: "Quit and restart the app." Either forces `link_subagents_post_processing` to re-run and set the flag.

**Explicit scope exclusion:** If, after this plan ships, a user reports a DB with many long-stale parent conversations where old-data "false-running" is a meaningful UX papercut, a follow-up quick task can add a trivial one-shot backfill: `UPDATE tool_calls SET subagent_link_attempted = 1 WHERE name IN ('Task','Agent') AND conversation_id IN (SELECT id FROM conversations WHERE updated_at &lt; '&lt;migration-date&gt;')`, gated on a `migrations_applied` row. That is **NOT in scope** for IMPR-7.
