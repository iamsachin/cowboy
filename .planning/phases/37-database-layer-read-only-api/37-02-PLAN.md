**1. overview (port getOverviewStats + computePeriodStats, lines 12-149):**
- Query params: DateRangeParams (from, to, agent)
- computePeriodStats: 4 queries in one db.call():
  a. Token totals: SUM of all token columns from token_usage INNER JOIN conversations (date filter + optional agent)
  b. Conversation count: COUNT(*) from conversations WHERE hasAssistant EXISTS subquery (same as Node.js)
  c. Active days: COUNT(DISTINCT date(created_at)) from conversations WHERE hasAssistant
  d. Per-model tokens: GROUP BY conversation_id, model for cost calculation via calculate_cost
- Prior period: compute dates using chrono -- prior_to = from_date - 1 day, prior_from = prior_to - period_length
- Trend calculation: ((current - prior) / prior) * 100, null if prior is 0
- Response struct OverviewStats with nested trends object (rename_all = "camelCase")
- CRITICAL: Append T23:59:59Z to the `to` date for inclusive range (match Node.js pattern)

**2. timeseries (port getTimeSeries, lines 155-223):**
- Query params: DateRangeParams + granularity: Option<String>
- Auto-granularity: if no granularity param, compute from date range (port the 14-line function):
  - < 14 days -> "daily", <= 90 days -> "weekly", else -> "monthly"
- Date format: daily=%Y-%m-%d, weekly=%Y-W%W, monthly=%Y-%m
- Query: GROUP BY strftime period + model, JOIN token_usage with conversations
- Post-process: aggregate per-model rows into per-period rows, summing costs via calculate_cost
- Response: Vec<TimeSeriesPoint> sorted by period

**3. model-distribution (port getModelDistribution, lines 229-257):**
- Query: GROUP BY token_usage.model, COUNT(DISTINCT conversation_id), SUM of all tokens
- JOIN conversations for date filter
- Response: Vec<ModelDistributionEntry> { model, count, totalTokens }

**4. filters (port getFilterOptions, lines 980-1004):**
- Two simple queries: SELECT DISTINCT project (non-null), SELECT DISTINCT agent
- Date-scoped
- Response: { projects: Vec<String>, agents: Vec<String> }

**5. tool-stats (port getToolStats, lines 813-847):**
- Query: GROUP BY tool_calls.name, COUNT(*), SUM with CASE WHEN for success/failure/unknown/rejected
- success condition: status = 'success' OR status = 'completed'
- failure: status = 'error'
- unknown: status IS NULL
- rejected: status = 'rejected'
- JOIN conversations for date/agent filter
- avgDuration and p95Duration always null (Node.js returns null -- JSONL format lacks timing data)
- Response: Vec<ToolStatsRow> ordered by total DESC

**6. heatmap (port getHeatmapData, lines 853-872):**
- Simple query: date(created_at) as date, COUNT(*) grouped by date
- Date + agent filter
- Response: Vec<HeatmapDay> { date, count }

**7. project-stats (port getProjectStats, lines 878-975):**
- Main query: GROUP BY conversations.project with LEFT JOIN token_usage, counts, SUM tokens, MAX created_at as lastActive
- Per project, TWO secondary queries (in same db.call closure):
  a. Per-model tokens for cost calculation: GROUP BY model, SUM tokens, calculate_cost for each
  b. Top models: GROUP BY model, COUNT(DISTINCT conversation_id), ORDER BY count DESC
- Project null -> "Unknown" string
- Response: Vec<ProjectStatsRow> { project, conversationCount, lastActive, totalTokens, totalCost, totalInput, totalOutput, totalCacheRead, totalCacheCreation, topModels: Vec<ProjectModelEntry> }
- IMPORTANT: The secondary queries run per-project. For the Rust port, collect all projects from the main query, then for each project run the two secondary queries inside the same db.call() closure to minimize thread hops.

**8. token-rate (port getTokenRate, lines 1011-1029):**
- Query: strftime('%Y-%m-%dT%H:%M', created_at) as minute, SUM input/output tokens
- WHERE created_at >= strftime('%Y-%m-%dT%H:%M:%S', 'now', '-60 minutes')
- No date params, no agent filter (all agents combined)
- Response: Vec<TokenRatePoint> { minute, inputTokens, outputTokens }
