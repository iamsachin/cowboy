    **1. Create NewMessagesPill.vue:**

    A floating pill component anchored to the bottom of the viewport. Positioned `fixed bottom-6 left-1/2 -translate-x-1/2 z-50`. Uses DaisyUI night theme styling: `btn btn-sm btn-primary gap-1 shadow-lg`. Shows "{count} new message{s}" with a ChevronDown icon. Emits `scrollToBottom` on click. Wrapped in a Vue `<Transition>` with a fade enter/leave animation. Only visible when `count > 0`.

    Props: `count: number`. Emits: `scrollToBottom`.

    **2. Modify ConversationDetail.vue:**

    Add new props from parent (ConversationDetailPage will pass these down):
    - `conversationId: string` — needed to detect conversation change vs live update
    - `newGroupKeys: Set<string>` — from useConversationDetail, used for fade-in marking
    - `scrollContainerRef: HTMLElement | null` — the scrollable ancestor element

    **Fix pagination reset (CRITICAL):**
    Replace the existing `watch(() => props.messages.length, ...)` that resets `visibleCount` to `PAGE_SIZE` with a watch on `props.conversationId`. This ensures pagination only resets when navigating to a different conversation, NOT when new messages arrive via live refetch.

    Also reset `focusedGroupIndex` on `conversationId` change (not messages.length).

    **Auto-expand on live update when at bottom:**
    Import and use `useScrollTracker(scrollContainerRef)`. When `isAtBottom` is true and messages change, set `visibleCount.value = turns.value.length` so all groups are visible (no "load more" button hiding new content). When NOT at bottom, keep `visibleCount` unchanged.

    **Scroll position management:**
    Before the data.value updates in the parent trigger a re-render, call `captureScrollPosition()` from useScrollTracker. After Vue's nextTick, the returned restore function adjusts scrollTop. Implementation: watch `props.messages` with `{ flush: 'pre' }` to capture before DOM update, then in the same watcher use `nextTick()` to restore after.

    Actually, the scroll capture/restore should be coordinated with the parent. The simpler approach: watch `props.messages` with `flush: 'post'` — after DOM update, if `isAtBottom` was true before the update, call `scrollToBottom()`. If not at bottom, the DOM update via keyed v-for should preserve position naturally (since existing elements keep their keys). Only if content ABOVE the viewport changes (rare with append-only) would explicit restoration be needed. Start with the auto-scroll-if-at-bottom approach and add explicit restoration only if testing reveals drift.

    **Fade-in animation for new groups:**
    Add a CSS class `group-fade-in` with a 200ms ease-out animation (opacity 0->1, translateY 4px->0). In the `v-for` loop, conditionally add this class: `:class="{ 'group-fade-in': newGroupKeys.has(turnKey(turn)) }"` on the wrapper div of each turn. After 200ms, the animation completes naturally (CSS animation, no cleanup needed). The `newGroupKeys` set is replaced on each refetch, so old "new" markers automatically clear.

    **New messages pill integration:**
    Compute `newMessageCount` as `newGroupKeys.size` (but only when user is NOT at bottom — if at bottom, auto-scroll handles it and pill is unnecessary). Place `<NewMessagesPill>` inside the template, passing the count. On `scrollToBottom` emit, call `scrollToBottom(true)` from useScrollTracker (smooth mode per user decision).

    **3. Ensure collapse state preservation:**
    The existing `useCollapseState` is keyed by group ID (first message ID via turnKey). Since we keep the same keying and do not reset collapse state on refetch, this should work automatically. Verify that `useCollapseState` is NOT reset anywhere on messages.length change. If it is, fix it to only reset on conversationId change.
